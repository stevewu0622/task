# Brainstorming & Architecture Guide

針對您提出的需求與問題，以下是我的詳細分析與建議：

## 1. 關於可行性與費用
> **Q: 製作這個app我不想花任何費用，是可以做到的嗎?**

**A: 是的，完全可以。**
*   **前端 hosting:** 可以使用 GitHub Pages, Vercel 或 Netlify 免費託管 React 網頁。
*   **後端/資料庫:** Google Sheets API 是免費的（在一定額度內，對於內部小團隊使用綽綽有餘）。
*   **通知:** 瀏覽器的 `Notification API` 是內建功能，完全免費。

## 2. 關於 App 形式
> **Q: 透過電腦上可以用的 Web App 好嗎?**

**A: 這是最好的選擇。**
*   開發原生 App (iOS/Android) 門檻高且發布麻煩。
*   **PWA (Progressive Web App):** 本次提供的程式碼可以作為網頁運行。您可以透過 Chrome 瀏覽器的 "安裝到電腦" 功能，把它像 App 一樣釘選在工具列，開啟時會有獨立視窗，體驗非常接近原生電腦 App。

## 3. 關於 Google Sheets 與 通知機制 (關鍵點)
> **Q: 同事那邊就會跳通知出來提醒... 基於 Google Sheet。**

這是一個技術上的 Trade-off (取捨)：
*   **Google Sheets 的特性:** 它是「被動」的資料庫。當有人寫入資料時，Sheet **不會**主動通知其他人的電腦。
*   **解決方案 (Polling 輪詢):** 
    *   App 需要每隔一段時間 (例如每 30 秒) 自動去檢查 Google Sheet 有沒有新的 Task。
    *   如果有新 Task，App 就會觸發電腦的桌面通知。
    *   **限制:** 通知不會是「毫秒級」的即時，會有幾秒到幾十秒的延遲，但對於工作指派來說通常足夠。
    *   **替代方案:** 如果您非常介意即時性，建議改用 **Firebase (Google 的另一個產品)**，它的免費方案支援 Real-time Database (WebSocket)，可以做到真正的秒跳通知。但 Google Sheets 的好處是數據可視化，您可以直接打開 Excel 表格看資料。

## 4. 安全性與申請機制
> **Q: 需要有一個先申請才能加入使用的動作。**

**實作邏輯 (已包含在本次程式碼中):**
1.  **註冊:** 新用戶輸入 Email/密碼，狀態標記為 `PENDING` (待審核)。
2.  **管理員:** 第一個註冊的人自動成為管理員 (Admin)，或者是寫死在設定檔中。
3.  **審核:** 管理員在 App 內會看到「申請列表」，點擊「核准」後，該用戶才能登入並看到任務。

## 5. 本次實作說明
為了讓您能立刻體驗 UI/UX，這個版本使用了 **瀏覽器本地資料庫 (LocalStorage)** 來模擬 Google Sheets 的行為。
*   您可以註冊、審核、指派任務、切換帳號測試。
*   **未來對接 Google Sheets:** 只需要修改 `services/storage.ts` 檔案，將 `localStorage` 的讀寫操作替換為 `fetch()` 呼叫 Google Sheets API (Google Apps Script) 即可。

## 功能亮點
*   ✅ **權限系統:** 登入、註冊、管理員審核。
*   ✅ **任務指派:** 支援多人指派、設定期限。
*   ✅ **狀態追蹤:** 待處理 -> 進行中 -> 已完成。
*   ✅ **桌面通知:** 模擬收到新任務時的系統通知。
*   ✅ **篩選與搜尋:** 針對同事、狀態、關鍵字進行篩選。
